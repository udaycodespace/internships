# Why Passwords Are Not Visible in CLI / Terminals

## The Main Reason

When you type a password in terminals like:

* WSL
* Linux Terminal
* PowerShell
* Azure CLI
* SSH
* Git
* Docker

the terminal intentionally hides the characters.

So even if you type:

```bash
mypassword123
```

the screen shows:

```text
(nothing)
```

But the input is still being received by the system.

---

> [!NOTE]
> This behavior is called **disabling terminal echo**.

---

# What is Terminal Echo?

Normally, terminals work like this:

```text
Keyboard → Terminal → Screen
```

Every key you type gets:

1. captured by the OS
2. sent to the application
3. echoed back to the screen

Example:

You type:

```text
hello
```

Terminal displays:

```text
hello
```

That display behavior is called:

```text
Terminal Echo
```

---

# What Happens During Password Input?

When a password prompt appears, the application temporarily disables echo.

Conceptually:

```c
disable_terminal_echo();
read_password();
enable_terminal_echo();
```

So internally:

```text
Keyboard → Application
```

still works,

but:

```text
Application → Screen
```

is blocked.

---

> [!IMPORTANT]
> Your keyboard input is still being captured.
>
> The terminal only stops displaying it.

---

# Why Hide Passwords Completely?

## Security Reasons

If passwords were fully visible:

```text
mypassword123
```

then anyone nearby could see them instantly.

This creates risks like:

* Shoulder surfing
* Screen recording leaks
* Streaming leaks
* Shared screen exposure
* Terminal history exposure

---

> [!CAUTION]
> Showing the actual password directly is considered unsafe in most secure systems.

---

# Then Why Not Show `*****` or `•••••`?

Good question.

Many modern apps do this:

```text
********
```

or:

```text
••••••••
```

instead of showing the real password.

But terminals often avoid even that.

---

# Problems With Showing `*****`

## 1. Password Length Leakage

Example:

```text
************
```

Now an attacker knows:

```text
Password length = 12
```

That already reduces guessing complexity.

---

## 2. Typing Pattern Analysis

Attackers can observe:

* typing speed
* pauses
* rhythm
* corrections/backspaces

This is called:

```text
Keystroke Dynamics
```

Example:

```text
**** ******* **
```

might hint at:

* spaces
* multiple words
* typing habits

---

## 3. Extra Terminal Complexity

Classic Unix terminals were extremely simple.

Displaying masking characters required:

* additional rendering
* cursor handling
* backspace synchronization

Old systems preferred:

```text
No echo at all
```

because it was simpler and safer.

---

> [!NOTE]
> Modern GUIs prioritize user reassurance.
>
> Traditional terminals prioritize minimalism and security.

---

# Why Not Add an Eye Icon Like Websites?

Websites often have:

```text
[ 👁 Show Password ]
```

But terminals usually do not.

---

# Reasons

## 1. CLI is Keyboard-Driven

Terminals are designed around:

* keyboard input
* text streams
* minimal UI rendering

An eye button requires:

* graphical UI
* mouse interaction
* dynamic rendering

Traditional terminals were never built for that.

---

## 2. Compatibility Across Systems

CLI tools run everywhere:

* SSH sessions
* remote servers
* containers
* embedded systems
* recovery terminals

A graphical eye icon would not work consistently across all environments.

---

## 3. Security Philosophy

CLI tools usually follow:

```text
Minimal UI
Minimal exposure
Minimal assumptions
```

The philosophy is:

> “Do not display sensitive information unless absolutely necessary.”

---

# Why It Sometimes Feels Like Bad UX

For beginners, this can feel confusing:

```text
"Am I typing?"
"Is the keyboard working?"
"Did the terminal freeze?"
```

So yes — from a beginner UX perspective, it can feel unfriendly.

But experienced CLI users expect this behavior immediately.

---

> [!IMPORTANT]
> In security systems, usability is sometimes intentionally reduced to improve safety.

---

# Quick Summary

| Behavior                      | Reason                                 |
| ----------------------------- | -------------------------------------- |
| Password invisible            | Terminal echo disabled                 |
| No `*` shown                  | Avoid leaking password length/patterns |
| No eye icon                   | CLI is text-based and minimal          |
| Input still works             | Characters are captured internally     |
| Common in WSL/Linux/Azure CLI | Standard Unix terminal behavior        |

---

# Real Example

## Normal Input

```bash
echo hello
```

Typing:

```text
hello
```

appears on screen.

---

## Password Input

```bash
sudo apt update
```

Prompt:

```text
[sudo] password for uday:
```

You type:

```text
mypassword123
```

Screen shows:

```text
(nothing)
```

But after pressing Enter, authentication still works.

---

# Final Core Idea

> [!TIP]
> Passwords are hidden in terminals because applications temporarily disable terminal echo to reduce password exposure and minimize security risks.
